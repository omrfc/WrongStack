## Finding 1 — Command Injection via PowerShell String Interpolation

**File:** `packages/runtime/src/clipboard.ts`  
**Lines:** 23–37  
**Category:** Command Injection (shell executions)  
**Severity:** HIGH  
**Confidence:** High

### Evidence

```typescript
// Line 23-37
async function readWindows(): Promise<ClipboardImage | null> {
  const tmp = path.join(os.tmpdir(), `wstack-clip-${Date.now()}.png`);
  const ps = [
    'Add-Type -AssemblyName System.Windows.Forms',
    'Add-Type -AssemblyName System.Drawing',
    '$img = [System.Windows.Forms.Clipboard]::GetImage()',
    'if ($img -eq $null) { Write-Output "NO_IMAGE"; exit 0 }',
    `$img.Save('${tmp.replace(/\\/g, '\\\\')}', [System.Drawing.Imaging.ImageFormat]::Png)`,
    'Write-Output "OK"',
  ].join('; ');
  const out = await runCmd('powershell', ['-NoProfile', '-Command', ps]);
```

The variable `tmp` (a file path under `os.tmpdir()`) is **interpolated directly into a PowerShell script string** and executed via `spawn`. While `tmp` is generated by `path.join(os.tmpdir(), ...)` and `os.tmpdir()` is a system directory controlled by the OS, the `replace(/\\/g, '\\\\')` is incomplete — it escapes backslashes for PowerShell string literals but does **not** sanitize single quotes. A path containing a single quote (e.g., a user-named temp directory) would break out of the string literal and allow arbitrary PowerShell code execution.

The same pattern appears in `readDarwin()` (lines 39–57), where the `tmp` path is interpolated into an AppleScript `open for access` command, with no quote sanitization.

### Why it's dangerous

An attacker who can influence the OS temp directory path (e.g., by setting `$TMPDIR` on macOS/Linux — though here it's system-controlled) or who has a user profile path containing special characters could inject shell commands. On Windows the attack surface is lower since `os.tmpdir()` resolves to a system path, but the code is still **not safe by construction** — it relies on environmental invariants rather than explicit validation.

### Recommendation

Use the `file://` URI scheme or pass arguments via the process environment instead of string interpolation, or use a strict allowlist of safe characters:

```typescript
// Validate tmp path contains only safe chars before interpolation
if (!/^[a-zA-Z0-9/_:-]+$/.test(tmp)) throw new Error('Unsafe clipboard path');
```

## Finding 2 — Sensitive Data Leaks to `console.warn` / `console.log`

**File:** `packages/core/src/security/secret-vault.ts`  
**Lines:** 129–134  
**Category:** Sensitive data exposure in logs  
**Severity:** MEDIUM  
**Confidence:** High

### Evidence

```typescript
// Line 129-134 (decryptConfigSecrets)
} catch (err) {
  console.warn(
    `[secret-vault] Failed to decrypt "${key}":`,
    err instanceof Error ? err.message : err,
  );
  return '';
}
```

The warning message includes the **key name** (e.g., `"apiKey"`, `"openai_key"`) and the **error message** from a decryption failure. If this runs in a shared terminal or gets printed to a log aggregator, it leaks the names of which secret fields exist in the config — even though the actual secret values are not printed, the **field names themselves** are sensitive metadata (they reveal which APIs the user has configured, whether they use Anthropic vs OpenAI, etc.).

The same pattern appears in:

- `secret-vault.ts` line 254: `console.warn('[secret-vault] Could not restrict permissions...')` — prints the path, which may contain usernames on Windows (`C:\Users\alice\.wrongstack\config.json`)
- `config-loader.ts` line 274: `console.warn('[config] Failed to read "${file}":', err)` — leaks file path  
- `config-loader.ts` line 283–285: `console.warn('[config] Failed to parse "${file}": invalid JSON.')` — leaks config file path
- `session-store.ts` line 170–173: logs session id on manifest write failure

### Why it's dangerous

In multi-tenant or shared environments, a `console.warn` with a key name like `"github_pat"` or `"aws_access_key"` in the output reveals that the user has configured a GitHub personal access token or AWS credentials — this is sensitive operational security data.

### Recommendation

Replace `console.warn` with the injected `Logger` interface (already used throughout the codebase) so the host controls log routing and can redact sensitive metadata:

```typescript
opts.logger.warn(`[secret-vault] Failed to decrypt field (name redacted)`);
```

## Finding 3 — Path SubjectKey Not Validated Against Trust File Scope

**File:** `packages/core/src/security/permission-policy.ts`  
**Lines:** 270–278  
**Category:** Improper input validation  
**Severity:** MEDIUM  
**Confidence:** Medium

### Evidence

```typescript
// Line 270-278
if (subjectKey) {
  const v = obj[subjectKey];
  if (typeof v === 'string') {
    const isPathKey = subjectKey === 'path' || subjectKey === 'file' || subjectKey === 'files';
    return isPathKey
      ? normalizePath(v)
      : escapeGlob(v);
  }
```

The code uses a string-comparison heuristic (`subjectKey === 'path' || subjectKey === 'file'`) to decide whether to treat a value as a filesystem path for glob-normalization. A tool with a field named `path` that actually means **URL path** (not filesystem path) will have its value fed through `normalizePath()` which converts `\` to `/` and escapes glob metacharacters — but the **trust file rules written for filesystem paths** will then be matched against this normalized value, potentially causing incorrect matches.

Conversely, a tool field named something other than `path`/`file`/`files` that contains a real filesystem path will have the path treated as an opaque string, preventing proper glob matching in the trust file.

### Why it's dangerous

Permission bypass — a path like `foo/bar/**` could be matched by an overly broad trust rule if the normalization creates unexpected glob interactions, or a legitimate file path could fail to match a properly specific trust rule because it wasn't recognized as a path.

### Recommendation

Make the path-vs-opaque decision based on an explicit declared property on the Tool definition rather than heuristic string matching:

```typescript
const isPathKey = tool.declaresPathSubject ?? (subjectKey === 'path' || subjectKey === 'file');
```

## Finding 4 — Glob Pattern Length Cap Only Enforced at Compile Time

**File:** `packages/core/src/utils/glob-match.ts`  
**Lines:** 33–38  
**Category:** Improper input validation  
**Severity:** LOW  
**Confidence:** Medium

### Evidence

```typescript
// Line 33-38
const MAX_GLOB_PATTERN_LEN = 1024;

export function compileGlob(pattern: string): RegExp {
  if (pattern.length > MAX_GLOB_PATTERN_LEN) {
    throw new Error(`Glob pattern exceeds ${MAX_GLOB_PATTERN_LEN} characters`);
  }
```

The length cap on glob patterns (`MAX_GLOB_PATTERN_LEN = 1024`) is only enforced inside `compileGlob()`, but the compiled regex is **cached** in `COMPILED_GLOB_CACHE`. An attacker (or a compromised tool definition) that submits a 1025-character pattern would get an exception on first compile, but if the same pattern is submitted again the cached regex (from before the limit was added) would be returned — bypassing the cap for repeated submissions.

Additionally, the cache eviction policy (`CACHE_MAX_SIZE = 2000`) could evict older entries, but there's no eviction of entries that exceed the length limit — a large pattern submitted once and cached would persist until 2000 other patterns accumulate.

### Why it's dangerous

ReDoS via cache exhaustion: an attacker submits many unique long patterns to fill the cache with compiled regexes that were valid before the cap was added, then a subsequent submission of a dangerous pattern that would have been caught by the length check could be served from cache, bypassing the validation.

### Recommendation

Move the length check to `getCachedGlob()` before cache lookup, or store the pattern length alongside the cached regex and validate on cache hit.

## Positive Security Observations (Not Vulnerabilities)

The following categories were thoroughly searched and **no vulnerabilities were found**:

### ✅ eval() / Function() — CLEAN
No `eval()`, `new Function()`, or `Function()` usage found anywhere in the codebase. Skill templates are parsed as Markdown frontmatter + text and are never eval'd.

### ✅ Path Traversal — CLEAN
`skill-installer.ts` (lines 93–96) has explicit path traversal detection:
```typescript
const resolved = path.resolve(destPath);
if (!resolved.startsWith(path.resolve(destDir))) {
  throw new Error(`Path traversal detected in skill file: ${file}`);
}
```
This is **good practice** and correctly implemented.

### ✅ SSRF — CLEAN
`github-fetcher.ts` fetches only `https://api.github.com/repos/...` — hardcoded domain, no user-controlled URL construction. The `fetch` call uses `redirect: 'follow'` but the final URL is pre-determined and the GitHub API is a trusted endpoint.

### ✅ Insecure Random — CLEAN
All random generation uses `node:crypto` (`randomBytes`, `randomUUID`) — no `Math.random()` used for any security-relevant purpose.

### ✅ Hardcoded Credentials — CLEAN
No API keys, tokens, or secrets found in source code. All credentials come from config files, environment variables, or the `SecretVault` AES-256-GCM encryption system.

### ✅ Insecure Deserialization — CLEAN
`safeParse()` in `safe-json.ts` has a 5MB size cap before calling `JSON.parse()`. Session store loads JSONL with explicit `type` and `ts` discriminator validation before pushing events.

### ✅ ReDoS — CLEAN
`regex-guard.ts` explicitly compiles user regexes with a `MAX_PATTERN_LEN = 512` and checks for catastrophic backtracking patterns (`DANGEROUS_PATTERNS`). The `compileGlob()` function builds regexes from patterns character-by-character without nested quantifiers.

### ✅ Prototype Pollution — CLEAN
Both `secret-vault.ts` (line 148: `Object.create(null)`) and `config-loader.ts` (line 100: `Object.create(null)`) use explicit null-prototype objects and check `FORBIDDEN_PROTO_KEYS` during deep merge. The `deepMerge` in `secret-vault.ts` explicitly skips `__proto__`, `constructor`, and `prototype`.

### ✅ Missing Security Headers — N/A
No HTTP server implementations found in the reviewed source. The OTLP exporter in `otlp-traces.ts` is a **client** (outbound POST), not a server. No Express/HTTP server patterns present.

### ✅ CORS Misconfiguration — N/A
No HTTP server implementations found. CORS is not relevant to this codebase.

### ✅ Auth/Authz Bypass — CLEAN
The `DefaultPermissionPolicy` has a well-structured evaluation chain with explicit deny/allow precedence. The `AutoApprovePermissionPolicy` for subagents explicitly respects `permission: 'deny'` tool defaults. The trust file mechanism uses `atomicWrite` to prevent torn writes.

### ✅ Secure Random for Secrets — CLEAN
`secret-vault.ts` uses `crypto.randomBytes(IV_BYTES)` for IVs and `crypto.randomBytes(KEY_BYTES)` for key generation. Key file is written with mode `0o600` on POSIX and `icacls` restriction on Windows.

### ✅ Child Process Credential Stripping — CLEAN
`child-env.ts` has a thorough allowlist approach — it copies only explicitly allowlisted vars (`PATH`, `HOME`, `LANG`, etc.) and strips anything matching secret name patterns (`TOKEN`, `SECRET`, `PASSWORD`, `AUTH`, etc.) from child process environments. A prominent opt-in warning is shown when passthrough is enabled.
